                                    Brilliant post (as always). One tiny pedantic nitpick: DeepQ does not "destroy" humans in Breakout, or even match a non-novice player.                                             You rightly pointed out that model-building will be key to scale up RL.

                                            IMHO, the deep generative modeling community (esp for vision data) is a bit lost or going in a different direction. I think building models that learn intermediate representations like objects, their intrinsic properties like relative pose/location, 3D shape etc will be crucial for RL and other tasks (mostly for solving goals that humans solve in physical environments). This will enable agents to explore hierarchically without wasting time on a senseless manifold. For this to work, we need to put in a lot more structure in our deep generative models -- spatio-temporal continuity, inductive biases for variable length items, and other inductive biases for perceptual binding. Will simply putting the right inductive structure be enough to bootstrap learning such structured representations? I don't know. We will probably need semi-supervised approaches to further bootstrap this and therefore probably need new datasets.

                                                Hey Tejas, thanks for dropping by :) I go back and forth about exactly how much structure we want to build into the networks.                                                           Aravind Srinivas L Tejas Kulkarni • 2 years ago

                                                          Hi Tejas,

                                                          I am not sure if we should really move away from generative models. Have a look at this work of ours https://arxiv.org/abs/1605.... - The framework is able to learn spatio-temporal abstractions and constructs options naturally to move between different abstract states, thereby giving us nice object oriented policies without having to explicitly let the agent know of the objects in the world.

                                                          We have practical issues such as the scalability of the components involved. But on our experiments on Mario and Seaquest, we were able to discover options connecting abstract states so as to learn skills like "kill the monster" , "get the key", etc or "go to the oxygen filler when low on oxygen or when you captured 6 humans" (seaquest).... This was done using a generative next frame predictive model, which is used to encode the representation for a frame. You can think of this as a model knowledge. As mentioned in the post, we do bring in our knowledge from physics models of bounce, etc while playing Pong. However, we still haven't addressed the sparse reward issues here... The framework needs some initial trajectories to learn a predictive model. For Montezeuma, we wanted to try with a set of human trajectories so that abstractions of useful regions of the state space are learnt. I believe this is a less supervised way than having a set of object oriented skills specific to the task known before hand.

                                                          However, the idea mentioned in the blog post of having a natural language set of rules is a good direction. You do know a set of rules that help you. Having an attention on the rules while playing games sounds like a cool idea.

                                                                                                ibmua Shalabh Singh • 6 months ago

                                                                                                Actually, after tinkering for two days, I do now see that Flappy indeed is totally unsuitable for the kind of PG that's described in this article. I first trained with frame skipping, so this wasn't much of an issue, but when I switched to per-frame policy checking it became clear that the kind of rewards described in this article are unsuitable for Flappy and many many other problems. For pong it may work well, but for Flappy it's only going to lead it to learn to get stuck at the very beginning choosing to go up on almost every frame (or all if not choosing stochastically).

                                                                                                For Flappy and I'd go as far as to say "in general" it's not a good idea to apply negative reinforcement. You should only teach from positive experience. I don't know if anybody else tried this, I'm guessing that yes, because it's a pretty obvious solution to me, but I haven't seen it anywhere and it seems pretty different from A3C, it's a better idea (than "vanilla PG") to train two outputs: one for answer prediction and another one for saying if you should better not listen to the answer provided and explore - make random choices, or use whatever your fallback algorithm is. I call it 🅴🆇🅿🅻🅾🆁🅰🆃🅸🅾🅽 🅽🅴🆄🆁🅾🅽. We shouldn't discourage a negative chain of actions, because we have no idea which of them were wrong. Instead we should not reinforce them at all and "mark" them so that when similar situations come it would tell us and we would know that our net has ɴᴏ ɪᴅᴇᴀ how to solve them. It's probably important in some cases to make a separate neural branch for this neuron as essentially you provide it with loss at times when you don't provide the decision neuron with it and that's about the only time that you tell it to up the value, so it might significantly rebuild the network that decision neuron is connected to.

                                                                                                I'm not sure this is the whole solution though. I think training on past, or someone else's 🅿🅾🆂🅸🆃🅸🆅🅴 experiences is also really crucial for 99.9% of AI applications. When doing experience replay it's probably a good idea to sometimes go through the memories and sort them for training by a value of difference between what the optimal strategy was and what the net chooses to do - that should help make it learn more efficiently and not overfit it. I'm guessing this is what these guys do https://arxiv.org/pdf/1511.... but I don't really know what "TD error" is. It might be a different thing, don't know.

                                                                                                Pong is just awfully easy, that's the only reason why simple PG described in this article works well enough to get the job done.
